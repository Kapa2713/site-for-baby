# site-for-baby

Проект для гендер-пати: красивый сайт, где гости делают игровые прогнозы на пол ребёнка — мальчик или девочка.

Важно: проект не принимает онлайн-платежи, не хранит реальные платёжные данные и не выплачивает реальные деньги. Это развлекательная механика для вечеринки. Денежные расчёты, если они вообще будут, проходят только офлайн между организаторами и гостями.

## Цель

Сделать простой MVP, который позволит:

1. Показать гостям красивую страницу с текущими коэффициентами на мальчика и девочку.
2. Перед прогнозом попросить гостя ввести имя и фамилию.
3. Принять игровой прогноз от гостя.
4. Сохранять, кто, сколько виртуальных денег поставил и на какой вариант.
5. Сохранять данные в Google Таблицу.
6. Автоматически пересчитывать коэффициенты после новых прогнозов.
7. Дать организаторам понятную таблицу со всеми прогнозами, докупками и общей статистикой.
8. На гендер-пати показать красивый итоговый экран.

## Формат продукта

Приоритет на старте — красивый сайт.

Основной вариант хранения данных — Google Таблица.

Предпочтительная схема для MVP:

```text
Сайт
→ Google Apps Script / небольшой backend-слой
→ Google Таблица
```

Сайт не должен хранить секреты и не должен напрямую отдавать гостям доступ к Google Таблице.

Telegram-бот — запасной вариант, если сайт с Google Таблицей окажется слишком сложным или долгим для первого релиза.

## Что входит в MVP

- публичная красивая страница с текущими коэффициентами;
- форма участника: имя + фамилия + выбор пола + сумма виртуальной ставки;
- сохранение прогноза в Google Таблицу;
- виртуальный баланс участника или ручная фиксация суммы ставки;
- история прогнозов;
- пересчёт коэффициентов после каждого прогноза;
- таблица для организаторов с прогнозами и статистикой;
- учёт докупок виртуальных денег;
- итоговая статистика: кто поставил, сколько, на какой вариант, сколько раз докупался;
- возможность закрыть приём прогнозов перед раскрытием пола;
- красивый экран для показа итогов на гендер-пати.

## Что не входит в MVP

- онлайн-оплата;
- банковские карты;
- криптовалюта;
- реальные выигрыши;
- сложная авторизация;
- мобильное приложение;
- Telegram-бот как обязательная часть первого релиза;
- сложная база данных, если Google Таблицы достаточно для события.

## Базовая логика коэффициентов

Коэффициенты должны меняться похоже на букмекерскую механику: если больше виртуальных денег поставили на один вариант, коэффициент на него снижается, а на противоположный вариант растёт.

Предлагаемая простая формула для MVP:

```text
seed = 1000
boy_score = boy_total + seed
girl_score = girl_total + seed
total_score = boy_score + girl_score
boy_odds = total_score / boy_score
girl_odds = total_score / girl_score
```

На старте оба коэффициента равны `2.00`. Чем больше ставят на мальчика, тем ниже коэффициент на мальчика и выше коэффициент на девочку. И наоборот.

Коэффициент фиксируется в момент прогноза, чтобы потом можно было показать, по какому коэффициенту человек сделал прогноз.

## Роли

### Гость

Может:

- открыть сайт;
- ввести имя и фамилию;
- посмотреть текущие коэффициенты;
- выбрать мальчика или девочку;
- указать сумму виртуальной ставки;
- отправить прогноз;
- увидеть, что прогноз принят.

### Организатор

Может:

- смотреть все прогнозы в Google Таблице или админ-представлении;
- смотреть суммы по мальчику и девочке;
- смотреть докупки;
- вручную добавлять докупку;
- закрывать приём прогнозов;
- видеть итоговую таблицу для вечеринки.

## Данные

Минимальные сущности:

- participant — участник;
- prediction — прогноз/ставка;
- top_up — докупка виртуальных денег;
- event_settings — настройки вечеринки;
- odds_snapshot — коэффициенты на момент ставки.

Минимальные поля участника:

```text
first_name
last_name
created_at
```

## Важные ограничения

- Не хранить реальные платёжные данные.
- Не использовать реальные деньги внутри приложения.
- Не обещать реальные выигрыши в интерфейсе.
- Не коммитить `.env` и секреты.
- Не хранить Google-ключи в frontend-коде.
- Не давать гостям прямой доступ к Google Таблице.
- Для настроек использовать `.env.example`, если настройки нужны.
- Не усложнять архитектуру без причины.

## Запасной вариант

Если сайт с Google Таблицей не получится быстро сделать, перейти на схему:

```text
Telegram-бот
→ Google Apps Script / backend-слой
→ Google Таблица
```

Бот использовать только как fallback, а не как первый вариант.
