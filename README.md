# site-for-baby

Проект для гендер-пати: красивый сайт, где гости делают игровые прогнозы на пол ребёнка — мальчик или девочка.

Важно: проект не принимает онлайн-платежи, не хранит реальные платёжные данные и не выплачивает реальные деньги. Это развлекательная механика для вечеринки. Денежные расчёты, если они вообще будут, проходят только офлайн между организаторами и гостями.

## Цель

Сделать простой MVP, который позволит:

1. Показать гостям красивую страницу с текущими предварительными коэффициентами на мальчика и девочку.
2. Показать гостям количество голосов за мальчика и девочку.
3. Перед прогнозом попросить гостя ввести имя и фамилию.
4. Принять игровой прогноз от гостя.
5. Сохранять, кто, сколько виртуальных денег поставил и на какой вариант.
6. Сохранять данные в Google Таблицу.
7. Автоматически пересчитывать коэффициенты после новых прогнозов.
8. Рассчитать итоговые выплаты после закрытия прогнозов и выбора результата.
9. Дать организаторам понятную таблицу со всеми прогнозами и итоговой статистикой.

## Формат продукта

Приоритет на старте — красивый сайт.

Основной вариант хранения данных — Google Таблица.

Предпочтительная схема для MVP:

```text
Сайт
→ Google Apps Script / небольшой backend-слой
→ Google Таблица
```

Сайт не должен хранить секреты и не должен напрямую отдавать гостям доступ к Google Таблице.

Telegram-бот — запасной вариант, если сайт с Google Таблицей окажется слишком сложным или долгим для первого релиза.

## Бюджет

Первый MVP должен быть сделан бесплатно или максимально близко к бесплатному варианту.

Требования:

- не использовать платный хостинг на старте;
- не использовать платную базу данных на старте;
- не покупать домен для первого MVP, если можно обойтись бесплатной ссылкой;
- использовать бесплатные лимиты и бесплатные инструменты, если их хватает для гендер-пати;
- если Codex предлагает платный сервис, он должен явно объяснить, зачем он нужен и есть ли бесплатная альтернатива;
- любое платное решение выносить как отдельное решение, а не добавлять автоматически.

Предпочтительные бесплатные варианты для MVP:

```text
GitHub Pages / другой бесплатный статический хостинг
Google Apps Script
Google Таблица
```

## Что входит в MVP

- публичная красивая страница с текущими предварительными коэффициентами;
- форма участника: имя + фамилия + выбор пола + виртуальная сумма;
- публичный блок с количеством голосов за мальчика и девочку;
- сохранение прогноза в Google Таблицу;
- история прогнозов;
- пересчёт коэффициентов после каждого прогноза;
- таблица для организаторов с прогнозами и статистикой;
- возможность закрыть приём прогнозов перед раскрытием пола;
- расчёт выплат по модели тотализатора;
- комиссия организатора, задаваемая в Google Таблице;
- итоговый лист `Results` с результатом каждого участника.

## Что не входит в MVP

- онлайн-оплата;
- банковские карты;
- криптовалюта;
- реальные выигрыши;
- сложная авторизация;
- мобильное приложение;
- отдельный display-экран;
- Telegram-бот как обязательная часть первого релиза;
- сложная база данных, если Google Таблицы достаточно для события;
- платные сервисы без отдельного согласования.

## Логика тотализатора

Комиссия задаётся в листе `Settings`:

```text
commission_percent = 10
```

Текущие суммы:

```text
total_pool = boy_total + girl_total
prize_pool = total_pool × (1 − commission_percent / 100)
```

Предварительные коэффициенты:

```text
boy_odds = prize_pool / boy_total
girl_odds = prize_pool / girl_total
```

Если ставок ещё нет, сайт показывает `2.00` для обоих вариантов. Если после начала события на один вариант ещё никто не поставил, для него показывается `—`.

Коэффициенты не фиксируют окончательную выплату. Поле `odds_at_bet` хранит только информационный снимок коэффициента сразу после принятия прогноза.

После закрытия прогнозов организатор задаёт:

```text
betting_open = false
result_gender = boy | girl
```

Затем вручную запускает функцию Apps Script:

```text
calculateSettlement
```

Если на победивший вариант есть ставки:

```text
final_odds = prize_pool / winner_pool
payout = winning_amount × final_odds
```

Комиссия организатора:

```text
organizer_commission = total_pool − prize_pool
```

Если на победивший вариант никто не поставил:

```text
prize_pool = 0
payout = 0
organizer_commission = total_pool
```

То есть организатор забирает весь виртуальный банк.

Результаты записываются:

- в дополнительные столбцы листа `Predictions` — результат каждой ставки;
- в лист `Results` — сводный итог по каждому участнику;
- в лист `Settings` — общий банк, призовой банк, комиссия и итоговый коэффициент.

## Роли

### Гость

Может:

- открыть сайт;
- ввести имя и фамилию;
- посмотреть текущие предварительные коэффициенты;
- посмотреть количество голосов за мальчика и девочку;
- выбрать мальчика или девочку;
- указать виртуальную сумму;
- отправить прогноз;
- увидеть, что прогноз принят.

### Организатор

Может:

- смотреть все прогнозы в Google Таблице;
- смотреть суммы по мальчику и девочке;
- смотреть количество голосов по мальчику и девочке;
- задавать комиссию;
- закрывать приём прогнозов;
- указывать итоговый пол;
- запускать окончательный расчёт;
- смотреть выплаты и чистый результат участников;
- смотреть итоговую комиссию организатора.

## Данные

Минимальные сущности:

- participant — участник;
- prediction — прогноз/ставка;
- event_settings — настройки вечеринки;
- odds_snapshot — предварительный коэффициент после ставки;
- settlement — итоговый расчёт;
- result — сводный результат участника.

Минимальные поля участника:

```text
first_name
last_name
created_at
```

Для публичного состояния сайта `getState` должен возвращать:

```text
bettingOpen
odds.boy
odds.girl
counts.boy
counts.girl
commissionPercent
```

`counts` — это количество прогнозов по каждому варианту, а не сумма виртуальных денег.

## Важные ограничения

- Не хранить реальные платёжные данные.
- Не использовать реальные деньги внутри приложения.
- Не обещать реальные выигрыши в интерфейсе.
- Не коммитить `.env` и секреты.
- Не хранить Google-ключи в frontend-коде.
- Не давать гостям прямой доступ к Google Таблице.
- Не добавлять платные сервисы без отдельного согласования.
- Для настроек использовать `.env.example`, если настройки нужны.
- Не усложнять архитектуру без причины.
- Перед расчётом итогов обязательно закрывать приём прогнозов.
- Для повторного расчёта сначала запускать `resetSettlement`.

## Запасной вариант

Если сайт с Google Таблицей не получится быстро сделать, перейти на схему:

```text
Telegram-бот
→ Google Apps Script / backend-слой
→ Google Таблица
```

Бот использовать только как fallback, а не как первый вариант.

## Следующий шаг

1. Перенести обновлённый `apps-script/Code.gs` в рабочий Google Apps Script.
2. Запустить `setupSpreadsheet`, чтобы создать новые настройки, столбцы и лист `Results`.
3. Проверить расчёт на текущих тестовых данных.
4. Опубликовать новую версию Google Apps Script Web App.
5. После проверки объединить ветку с `main`.
