# AGENTS.md

Этот репозиторий содержит MVP для гендер-пати: сайт, где гости делают игровые прогнозы на пол ребёнка, а коэффициенты на мальчика и девочку меняются в зависимости от распределения виртуальных ставок.

Главная цель проекта — быстро собрать понятный и рабочий прототип без лишней архитектуры. После технического MVP отдельный приоритет — довести публичный сайт до приятного mobile-first дизайна.

## 1. Что прочитать перед работой

Перед изменениями AI-агент должен прочитать:

1. `README.md`
2. `AGENTS.md`
3. `docs/prototype-brief.md`, если файл есть
4. `.env.example`, если проект использует настройки окружения
5. связанные файлы перед изменением кода

Если какого-то файла нет, агент должен прямо указать это.

## 2. Как работать с проектом

AI-агент должен:

- сначала понять задачу;
- определить тип задачи;
- изучить структуру проекта;
- найти связанные файлы;
- выбрать только нужные правила и чек-листы;
- предложить короткий план;
- делать минимальные осмысленные изменения;
- не переписывать весь проект без причины;
- не менять публичные интерфейсы без объяснения;
- не добавлять зависимости без причины;
- после изменений объяснять, что изменено и зачем.

Если пользователь прямо пишет «пока не кодь», «на этапе обсуждения» или просит сначала план — код не писать и файлы проекта не менять, пока план не согласован.

## 3. Как использовать ai-instructions

Если для задачи нужны общие правила, чек-листы, шаблоны или внешний каталог источников, использовать репозиторий:

```text

```

Не нужно читать все инструкции и источники подряд.

## 4. Тип проекта

На старте проект считать простым full-stack MVP.

Приоритетный формат — сайт.

Telegram-бот не делать первым шагом, пока отдельно не принято решение заменить сайт на бота или добавить бота вторым каналом.

Текущий технический стек должен оставаться простым:

```text
GitHub Pages
HTML + CSS + vanilla JS
Google Apps Script
Google Таблица
```

Не добавлять npm, Vite, Webpack, React, backend-хостинг, платные сервисы или тяжёлые animation-библиотеки без отдельного согласования.

## 5. Продуктовые ограничения

Проект не должен:

- принимать онлайн-платежи;
- хранить данные банковских карт;
- работать с реальными платёжными системами;
- обещать реальные выигрыши;
- использовать реальные деньги внутри приложения;
- выглядеть как полноценный букмекерский сервис.

Интерфейс должен объяснять механику как развлекательный прогноз для гендер-пати.

## 6. Данные и приватность

Минимально нужны:

- имя участника;
- фамилия участника;
- выбранный вариант: мальчик или девочка;
- сумма виртуальной ставки;
- коэффициент на момент ставки;
- дата и время ставки;
- докупки виртуального баланса, если они используются;
- статус события: ставки открыты или закрыты.

Не собирать лишние персональные данные.

Не логировать чувствительные данные.

Display-экран и публичные элементы сайта не должны показывать фамилии гостей и полный список прогнозов, если это отдельно не согласовано.

## 7. Безопасность

Если задача связана с API, пользователями, ролями, базой данных, секретами или AI-generated code, использовать:

```text
adapted/my-security-rules.md
checklists/security-checklist.md
```

AI-агент не должен:

- хранить секреты в коде;
- коммитить `.env`;
- логировать токены, пароли и ключи;
- отключать проверки безопасности;
- смешивать authentication и authorization;
- добавлять небезопасные зависимости без объяснения;
- принимать AI-generated code только потому, что он работает.

Минимальные правила:

- реальные секреты не коммитить;
- использовать `.env.example`;
- секреты передавать через environment variables / Script Properties;
- пользовательский ввод валидировать;
- права доступа проверять явно;
- лишние данные не отдавать наружу;
- чувствительные данные не логировать.

## 8. Frontend, дизайн и mobile-first

Для задач редизайна и улучшения публичного сайта использовать:

```text
sources/frontend-design-sources.md
adapted/my-frontend-design-rules.md
checklists/frontend-design-checklist.md
```

Целевой стиль сайта:

```text
нежный премиальный стиль + лёгкие игровые акценты
```

Целевая логика визуальных состояний:

```text
нейтральное состояние → мягкий общий фон
выбран мальчик       → голубой акцент + маленький эндуро-мотоциклист
выбрана девочка      → розовый акцент + маленькая принцесса
```

Анимация:

```text
средняя: плавные переходы, короткое появление тематического элемента, без перегруза и без тормозов на телефоне
```

Приоритеты frontend-доработки:

1. Mobile-first: сайт должен удобно открываться и заполняться с телефона.
2. Цельный фон: не должно быть резкого разрыва между верхом и низом страницы.
3. Понятная форма: минимум лишнего текста, крупные поля и кнопка.
4. Умеренная анимация: эмоция есть, но форма не страдает.
5. Быстрая загрузка: не добавлять тяжёлые зависимости и большие файлы.
6. Безопасность: дизайн-правки не должны ломать Apps Script, валидацию, EVENT_CODE и Google Sheets-интеграцию.

## 9. Humanizer и стиль текста

Если задача связана с README, документацией, интерфейсными текстами или объяснениями для гостей, использовать:

```text
adapted/my-humanizer-rules.md
```

Правила:

- сохранять смысл и факты;
- убирать AI-шные формулировки;
- убирать рекламность и канцелярит;
- писать проще, прямее и живее;
- не добавлять факты от себя;
- не делать технический текст разговорным в ущерб точности.

Если задача техническая, приоритет такой:

```text
точность → проверяемость → понятность → стиль
```

## 10. MVP

Первый MVP должен решать один сценарий:

```text
гость открыл сайт → увидел коэффициенты → ввёл имя и фамилию → ввёл код события → выбрал пол → указал виртуальную сумму → подтвердил прогноз → организатор увидел запись в Google Таблице
```

Не добавлять сложные функции, пока этот сценарий не работает.

Сейчас основной сценарий уже работает. Следующий фокус — привести публичный сайт к нормальному дизайну и мобильному виду без поломки рабочего сценария.

## 11. Тесты и проверки

Если есть тесты, запустить их.

Если тестов нет, предложить минимальные проверки:

```text
проверка основного сценария гостя
проверка Google Sheets записи
проверка пересчёта коэффициентов
проверка запрета ставки после закрытия приёма прогнозов
проверка .env.example
проверка security checklist
проверка frontend-design checklist
проверка мобильной версии 360–430px
проверка пользовательских текстов
```

Если проверку нельзя выполнить, агент должен прямо написать почему.

## 12. Формат ответа после работы

После изменений AI-агент должен написать:

1. Что сделано.
2. Какие файлы изменены.
3. Почему изменения нужны.
4. Какие инструкции и источники использованы.
5. Какие проверки выполнены.
6. Какие риски остались.
7. Что делать дальше.

## 13. Главное правило

Приоритет:

```text
понятно → проверено → применимо → без лишней архитектуры
```
