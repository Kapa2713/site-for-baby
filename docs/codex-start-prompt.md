# Задание для Codex

Работай в репозитории:

```text

```

Нужно спланировать первый MVP проекта `site-for-baby`.

## 1. Контекст проекта

У организаторов скоро гендер-пати. Нужен красивый сайт, где гости смогут делать игровые прогнозы на пол ребёнка: мальчик или девочка.

Сайт должен работать как развлекательная механика с виртуальными ставками. Он не должен принимать онлайн-платежи, хранить реальные платёжные данные, обещать реальные выигрыши или выглядеть как полноценный букмекерский сервис.

Организаторы хотят видеть, кто сделал прогноз, сколько виртуальных денег поставил, на какой вариант и сколько раз человек докупался.

## 2. Главное ограничение по бюджету

Первый MVP нужно спланировать бесплатно или максимально близко к бесплатному варианту.

Нельзя добавлять без отдельного согласования:

- платный хостинг;
- платную базу данных;
- платный домен;
- платные внешние сервисы;
- сложную инфраструктуру.

Предпочтительные бесплатные варианты:

```text
GitHub Pages / другой бесплатный статический хостинг
Google Apps Script
Google Таблица
```

Если предлагаешь платный сервис, обязательно объясни:

1. зачем он нужен;
2. можно ли без него;
3. какая есть бесплатная альтернатива;
4. почему бесплатной альтернативы недостаточно.

## 3. Приоритетное техническое направление

Основной вариант:

```text
Сайт
→ Google Apps Script / небольшой backend-слой
→ Google Таблица
```

Почему выбран сайт:

- хочется красивый визуальный интерфейс;
- гостям достаточно открыть ссылку;
- перед голосованием можно попросить имя и фамилию;
- организаторам удобно смотреть данные в Google Таблице;
- для маленького события не нужна тяжёлая база данных.

Запасной вариант:

```text
Telegram-бот
→ Google Apps Script / backend-слой
→ Google Таблица
```

Telegram-бот не делай первым шагом. Его нужно рассматривать только как fallback, если сайт с Google Таблицей окажется слишком сложным или долгим.

## 6. Основной сценарий MVP

MVP должен закрывать сценарий:

```text
гость открыл сайт
→ увидел коэффициенты
→ ввёл имя и фамилию
→ выбрал мальчика или девочку
→ указал виртуальную сумму
→ подтвердил прогноз
→ прогноз сохранился в Google Таблицу
→ коэффициенты пересчитались
→ организатор увидел запись в Google Таблице или админ-представлении
```

## 7. Что должно быть в MVP

- публичная красивая страница с текущими коэффициентами;
- форма прогноза для гостя;
- поля `имя` и `фамилия` перед отправкой прогноза;
- выбор варианта: мальчик или девочка;
- ввод суммы виртуальной ставки;
- сохранение прогноза в Google Таблицу;
- пересчёт коэффициентов после нового прогноза;
- фиксация коэффициента на момент прогноза;
- таблица или админ-представление для организаторов;
- список всех прогнозов;
- суммы по мальчику и девочке;
- список докупок виртуальных денег;
- возможность вручную добавить докупку;
- возможность закрыть приём прогнозов;
- итоговая статистика для показа на вечеринке.

## 8. Что не делать в MVP

- онлайн-оплату;
- банковские карты;
- криптовалюту;
- реальные выигрыши;
- сложную авторизацию;
- мобильное приложение;
- Telegram-бот первым шагом;
- сложную аналитику;
- тяжёлую архитектуру;
- полноценную базу данных, если Google Таблицы достаточно для события;
- платные сервисы без отдельного согласования.

## 9. Формула коэффициентов

Для первого MVP использовать простую формулу:

```text
seed = 1000
boy_score = boy_total + seed
girl_score = girl_total + seed
total_score = boy_score + girl_score
boy_odds = total_score / boy_score
girl_odds = total_score / girl_score
```

Правила:

- на старте коэффициенты `2.00 / 2.00`;
- коэффициенты округлять до 2 знаков;
- коэффициент фиксировать в момент ставки;
- если больше ставят на один вариант, его коэффициент снижается;
- расчёт не связан с реальными выплатами.

Если считаешь, что формулу лучше упростить или изменить, сначала объясни почему. Не меняй её молча.

## 11. Безопасность

Обязательные ограничения:

- не коммитить `.env`;
- создать `.env.example`, если нужны настройки;
- не хранить реальные секреты в коде;
- не хранить Google-ключи во frontend-коде;
- не давать гостям прямой доступ к Google Таблице;
- валидировать пользовательский ввод;
- не логировать токены, пароли, ключи и лишние персональные данные;
- разделять authentication и authorization;
- не отдавать гостям админские данные.

Frontend может отправлять данные только в безопасный промежуточный слой, например Google Apps Script Web App или другой согласованный backend.
