## 1. Текущий статус проекта

```text
GitHub Pages

→ закрытая Google Таблица
```

- прогноз записывается в Google Таблицу;

Сейчас задача не в backend и не в Google Таблице. Задача — улучшить дизайн, mobile-first UX и визуальные состояния.

## 4. Что не нравится в текущем дизайне

Текущий сайт технически работает, но визуально выглядит сыро.

Проблемы:

1. Страница визуально делится на две части: верх яркий, низ слишком бледный.
2. Нет цельной дизайн-системы: фон, карточки, форма и нижний блок выглядят раздельно.
3. Нужно лучше адаптировать сайт под телефон, потому что большинство гостей будут открывать его с телефона.
4. Нужно добавить эмоциональную реакцию на выбор пола.
5. Нужно сохранить простоту и не сломать рабочую интеграцию с Google Таблицей.

## 5. Целевой стиль

Стиль:

```text
нежный премиальный + лёгкие игровые акценты
```

Не делать:

- кислотные цвета;
- казино/букмекерский стиль;
- агрессивный спорт-дизайн;
- слишком детский дешёвый шаблон;
- перегруженные анимации;
- тяжёлые frontend-библиотеки.

## 6. Mobile-first

Проектировать сначала телефонную версию.

На телефоне должно быть:

- один понятный вертикальный поток;
- крупный заголовок;
- компактные коэффициенты;
- удобная форма;
- крупная кнопка;
- без горизонтального скролла;
- всё читается без приближения;
- сайт нормально смотрится на ширине 360–430px.

Desktop-версия должна быть расширением мобильной, а не наоборот.

## 7. Визуальные состояния при выборе пола

Нужно предложить мягкую реакцию интерфейса на выбор пола.

### Нейтральное состояние

До выбора:

```text
мягкий общий праздничный фон
нейтральная палитра
форма и коэффициенты выглядят цельно
```

### Если выбран мальчик

```text
голубой акцент
маленький мотоциклист на эндуро мотоцикле
мягкая анимация появления или выезда
не агрессивно, не страшно, не реалистичный опасный мотокросс
милый праздничный образ
```

### Если выбрана девочка

```text
розовый / персиковый акцент
маленькая принцесса
мягкая анимация появления
сказочный, но не перегруженный стиль
```

## 8. Анимация

Уровень анимации: средний.

Хорошо:

- плавная смена оттенка фона;
- короткое появление тематического элемента;
- glow или мягкий акцент выбранной карточки;
- микроанимация выбора;
- длительность примерно 200–900 мс.

Плохо:

- тяжёлые бесконечные анимации;
- большие видео;
- внешние animation-библиотеки без необходимости;
- элементы, которые закрывают форму;
- анимация, которая тормозит на телефоне.

## 9. Технические ограничения

Сохранять простой стек:

```text
HTML
CSS
vanilla JS
GitHub Pages
Google Apps Script
Google Таблица
```

Не добавлять без отдельного согласования:

- npm;
- Vite;
- Webpack;
- React;
- Vue;
- Tailwind;
- Bootstrap;
- backend-хостинг;
- платные сервисы;
- внешние CDN;
- analytics/tracking;
- тяжёлые animation-библиотеки.

Если предлагаешь новый asset, сначала объясни:

- откуда он будет взят;
- можно ли его использовать публично;
- какой размер файла;
- не будет ли он тормозить на телефоне;
- есть ли fallback без картинки.

## 10. Безопасность

Дизайн-изменения не должны ломать безопасность.

Запрещено:

- добавлять секреты во frontend;
- добавлять `EVENT_CODE` в GitHub;
- добавлять `SPREADSHEET_ID` в frontend;
- менять Apps Script без причины;
- отключать валидацию формы;
- показывать фамилии гостей на display-экране;
- отдавать полный список прогнозов в публичный интерфейс.
