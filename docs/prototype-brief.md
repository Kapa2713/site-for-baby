# Бриф прототипа: сайт для гендер-пати

## 1. Название прототипа

`site-for-baby`

## 2. Цель

Проверить красивый онлайн-формат для гендер-пати: гости заходят на сайт, вводят имя и фамилию, делают игровые прогнозы на пол ребёнка, а организаторы видят статистику по прогнозам, виртуальным ставкам, докупкам и текущим коэффициентам.

Проект не должен работать с реальными онлайн-деньгами. Внутри приложения используются только виртуальные суммы.

## 3. Пользователи

- Гости гендер-пати.
- Организаторы.

## 4. Основной сценарий

1. Организатор открывает приём прогнозов.
2. Гость получает ссылку на сайт.
3. Гость видит красивую страницу с текущими коэффициентами на мальчика и девочку.
4. Гость вводит имя и фамилию.
5. Гость выбирает вариант: мальчик или девочка.
6. Гость указывает сумму виртуальной ставки.
7. Сайт отправляет данные через безопасный промежуточный слой.
8. Прогноз сохраняется в Google Таблицу.
9. Сайт фиксирует коэффициент на момент прогноза.
10. Коэффициенты автоматически пересчитываются.
11. Организатор видит прогноз в Google Таблице или админ-представлении.
12. Перед раскрытием пола организатор закрывает приём прогнозов.
13. На вечеринке организатор показывает итоговую статистику.

## 5. Что входит в MVP

- Публичная красивая страница с текущими коэффициентами.
- Форма прогноза для гостя.
- Поля `имя` и `фамилия` перед отправкой прогноза.
- Сохранение прогноза в Google Таблицу.
- Пересчёт коэффициентов после нового прогноза.
- Таблица или админ-представление для организаторов.
- Список всех прогнозов.
- Суммы по мальчику и девочке.
- Список докупок виртуальных денег.
- Возможность вручную добавить докупку.
- Возможность закрыть приём прогнозов.
- Итоговая статистика для показа на гендер-пати.

## 6. Что не входит в MVP

- Онлайн-оплата.
- Интеграция с банками.
- Интеграция с платёжными системами.
- Реальные выигрыши.
- Сложная авторизация.
- Мобильное приложение.
- Telegram-бот как обязательная часть первого релиза.
- Сложная аналитика.
- Реферальная система.
- Полноценная база данных, если Google Таблицы достаточно для события.
- Платные сервисы без отдельного согласования.

## 7. Бюджет

Первый MVP должен быть сделан бесплатно или максимально близко к бесплатному варианту.

Codex должен планировать решение от бесплатных инструментов:

```text
GitHub Pages / другой бесплатный статический хостинг
Google Apps Script
Google Таблица
```

Ограничения:

- не использовать платный хостинг на старте;
- не использовать платную базу данных на старте;
- не покупать домен для первого MVP, если можно обойтись бесплатной ссылкой;
- не добавлять платные сервисы без отдельного решения организатора;
- если предлагается платный сервис, указать бесплатную альтернативу и объяснить, почему платный вариант вообще нужен.

## 8. Техническое направление

Приоритетная схема:

```text
Сайт
→ Google Apps Script / небольшой backend-слой
→ Google Таблица
```

Причина:

- сайт можно сделать красивым;
- гостям достаточно открыть ссылку;
- Google Таблица удобна организаторам;
- для маленького события не нужна тяжёлая база данных;
- можно быстро проверить механику.

Важное ограничение:

```text
Frontend не должен хранить секреты Google и не должен напрямую давать гостям доступ к Google Таблице.
```

Запасной вариант:

```text
Telegram-бот
→ Google Apps Script / backend-слой
→ Google Таблица
```

Telegram-бот рассматривать только если сайт с Google Таблицей окажется слишком сложным или долгим.

## 9. Данные

| Сущность | Поля | Источник | Кто заполняет | Комментарий |
|---|---|---|---|---|
| participant | id, first_name, last_name, created_at | форма сайта | гость | На старте достаточно имени и фамилии. |
| prediction | id, participant_id, gender, amount, odds_at_bet, created_at, source | форма сайта | гость | gender: `boy` или `girl`; source: `site`. |
| top_up | id, participant_id, amount, created_at, comment | Google Таблица / админка | организатор | Нужно видеть, сколько раз человек докупался. |
| event_settings | betting_open, seed_amount, created_at, updated_at | Google Таблица / настройки | организатор | `betting_open` закрывает приём прогнозов. |
| odds_snapshot | boy_total, girl_total, boy_odds, girl_odds, calculated_at | расчёт системы | система | Можно хранить историю или считать текущее значение на лету. |

## 11. Роли

| Роль | Что видит | Что может делать | Ограничения |
|---|---|---|---|
| Гость | текущие коэффициенты, форму прогноза, подтверждение прогноза | сделать прогноз | не видит полную админскую таблицу |
| Организатор | все прогнозы, докупки, суммы, итоговую статистику | закрыть ставки, добавить докупку, смотреть таблицы | доступ должен быть ограничен |

## 12. Статусы

| Статус | Что означает | Кто меняет | Следующий статус |
|---|---|---|---|
| betting_open | прогнозы принимаются | организатор | betting_closed |
| betting_closed | прогнозы закрыты | организатор | reveal_done |
| reveal_done | пол раскрыт, можно показывать финальную статистику | организатор | финальное состояние |

## 13. Формула коэффициентов

Для MVP использовать простую формулу со сглаживанием:

```text
seed = 1000
boy_score = boy_total + seed
girl_score = girl_total + seed
total_score = boy_score + girl_score
boy_odds = total_score / boy_score
girl_odds = total_score / girl_score
```

Где:

- `boy_total` — сумма виртуальных ставок на мальчика;
- `girl_total` — сумма виртуальных ставок на девочку;
- `seed` — стартовое сглаживание, чтобы на старте коэффициенты были `2.00 / 2.00` и не было деления на ноль.

Правила:

- коэффициенты округлять до 2 знаков;
- коэффициент фиксировать в момент ставки;
- если больше ставят на мальчика, коэффициент на мальчика снижается;
- если меньше ставят на девочку, коэффициент на девочку растёт;
- расчёт не должен быть связан с реальными выплатами.

## 14. Контрольные точки

Проверить:

- сайт открывается по ссылке;
- гость может сделать прогноз;
- нельзя сделать прогноз без имени;
- нельзя сделать прогноз без фамилии;
- нельзя сделать прогноз с отрицательной или нулевой суммой;
- нельзя сделать прогноз после закрытия приёма;
- прогноз сохраняется в Google Таблицу;
- коэффициенты пересчитываются корректно;
- коэффициент сохраняется на момент прогноза;
- организатор видит все прогнозы;
- организатор видит докупки;
- организатор может закрыть приём прогнозов;
- `.env` не коммитится;
- `.env.example` есть, если нужны настройки;
- пользовательский ввод валидируется;
- гости не получают доступ к служебным данным;
- MVP не требует платных сервисов для запуска.

## 15. Критерии готовности

Прототип готов, если:

- сайт можно открыть по ссылке;
- гость может ввести имя и фамилию;
- гость может отправить прогноз;
- прогноз сохраняется в Google Таблицу;
- коэффициенты меняются после прогноза;
- организатор видит таблицу прогнозов;
- организатор видит суммы по двум вариантам;
- организатор может добавить докупку;
- организатор может закрыть приём прогнозов;
- нет онлайн-платежей и реальных денежных операций;
- MVP можно запустить на бесплатных инструментах;
- в репозитории нет секретов.

## 16. Риски

- Если использовать реальные деньги в интерфейсе, проект может выглядеть как азартная игра.
- Если дать гостям прямой доступ к Google Таблице, они смогут увидеть или изменить лишние данные.
- Если хранить Google-ключи во frontend-коде, секреты будут доступны пользователям.
- Если не ограничить админку, гости смогут увидеть служебные данные.
- Если не фиксировать коэффициент на момент ставки, потом будет непонятно, кто по какому коэффициенту поставил.
- Если хранить данные только в браузере, организатор не увидит ставки всех гостей.
- Если выбрать слишком сложный стек, MVP может затянуться.
- Если выбрать платный сервис на старте, проект станет сложнее и дороже без явной необходимости.

## 17. Первый технический шаг

Codex должен сначала предложить план и стек, не писать код сразу.

Минимальный ожидаемый результат от Codex:

```text
1. Как он понял проект.
2. Какой бесплатный или условно бесплатный стек предлагает и почему.
3. Как сайт будет сохранять данные в Google Таблицу.
4. Какие файлы создаст первыми.
5. Как будет хранить данные.
6. Как будет считать коэффициенты.
7. Как ограничит доступ к админским данным.
8. Какие риски видит.
9. Какой первый маленький шаг предлагает.
```
